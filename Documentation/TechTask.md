# Введение
## Наименование программы
Наименование разрабатываемого программного обеспечения - **"Клиентское приложение для просмотра и редактирования рабочих задач".**

## Краткая характеристика области применения
Программное обеспечение предназначено для управления коллективным планированием задач: совместного (или автономного) просмотра, создания, удаления и редактирования задач, задания сроков выполнения и структурных связей между задачами (родитель-потомок).

# Назначение разработки
## Функциональное назначение
Программное обеспечение предназначено для управления совместным планированием задач потенциальными пользователями (например, сотрудниками компаний-разработчиков ПО) внутри рабочих групп и налаживания взаимодействия между ними в ходе рабочих процессов. Пользователи могут использовать клиентское приложения для создания рабочих календарей задач и отслеживания плана их выполнения.

## Эксплуатационное назначение
Программное обеспечение является инструментом, который помогает рабочей группе управлять проектами и рабочими процессами, а также отслеживать задачи и выстраивать коммуникации в команде.
Список рабочих задач становится доступным с любого рабочего компьютера, каждый сотрудник может открыть список задач в удобное для него время. Таким образом повышается уровень взаимодействия между работниками и руководством.

# Требования к программе или программному изделию
## Требования к функциональным характеристикам
### Требования к составу выполняемых функций
Программное обеспечение имеет следующие требования к функциональным характеристикам:
1. Вся информация о задачах должна храниться внутри базы данных, которая представляет из себя сервер календарей

2. В клиенте должна быть организована авторизация пользователей через email и пароль  
Аккаунт с соответствующими почтой и паролем должен быть заранее создан в сервисе, на котором находится сервер календарей.

3. Пользователь должен иметь следующие возможности при работе с клиентом:
    * входить в аккаунт;
    * иметь возможность выбрать источник, откуда он хочет загрузить календарь
    * работать в режиме оффлайн на клиенте;
    
    Вся работа на клиенте после первой синхронизации с сервером ведется в режиме оффлайн, в это же время все изменения хранятся на стороне клиента. Попытка синхронизации данных с сервером выполняется по нажатию соответствующей кнопки в пользовательском интерфейсе.

    * смена темы (светлая, темная);

   При авторизации пользователя необходимо подгружать информацию из сервера календарей и все задачи (todo). 

4. В программном обеспечении должна быть предусмотрена структура задач - дерево задач. В дереве задач необходимо реализовать отображение дополнительных столбцов с атрибутами задач (например, сроки начала и окончания, приоритет, статус: выполнена\не выполнена).

5. Пользователь должен иметь возможность работы с карточками задач и управления списком календарей (например, возможность добавления календарей с разных серверов).

6. Пользователь должен иметь возможность выбрать, какая версия данных, в случае конфликтов записей, является приоритетней - версия сервера или клиента.

7. Реализовать возможность сохранения логина и пароля пользователя в настройках.
    
### Требования к организации входных данных
Программное обеспечение получает на вход данные учетной записи пользователя (адрес электронной почты и пароль), которая была создана на стороне сервера. После выполняется проверка на корректность и правильность введенной информации. При успешной проверке загружаются с соответствующего сервера доступные пользователю календари.

### Требования к организации выходных данных
Выходными данными является работа с задачами на клиенте, которые синхронизируются с сервером

### Требования к временных характеристикам
Обеспечить минимальное время отклика при работе с системой. При установке соединения с облачным хранилищем установить ограничение на время ожидания ответа в размере `30 секунд`.

## Требования к надежности
Вероятность безотказной работы программы должна составлять не менее 99.99% при условии исправности оборудования и электросети.

### Отказы из-за некорректных действий оператора
Отказы программы возможны вследствие некорректных действий оператора (пользователя) при взаимодействии с операционной системой. Во избежание возникновения отказов программы по указанной выше причине следует обеспечить работу конечного пользователя без предоставления ему административных привилегий. Также перед работой с программой оператору необходимо ознакомиться с документом `«Программа TaskPlanner`. Руководство оператора (**ГОСТ 19.505-79**);

## Условия эксплуатации
Климатические условия эксплуатации, при которых должны обеспечиваться заданные характеристики, должны удовлетворять требованиям, предъявляемым к техническим средствам, которые касаются условий их эксплуатации.

Минимальное количество персонала требуемое для работы программы должно составлять не менее 3 штатных единиц - системный программист, оператор технической поддержки, пользователь.
Оператор технической поддержки и системный программист должны иметь техническое образование. 
Пользователь программного обеспечения должен иметь начальный уровень владения ПК.

## Требования к составу и параметрам технических средств
Для полноценной работы с разрабатываемым программным обеспечением требуется ноутбук или персональный компьютер в стандартной комплектации с операционными системами Windows, Linux.

## Требования к информационной и программной совместимости
### Требования к исходным кодам и языкам программирования
Исходные коды программы должны быть реализованы на языке Python. 

Серверная часть программы должна быть реализована на базе набора клиент-серверных программ для создания и использования хранилища данных -  например, Nextcloud.

Взаимодействие с сервером календарей должно быть организовано с помощью протокола CalDAV.

Пользовательский интерфейс должен быть создан в кроссплатформенной свободной среде для разработки графических интерфейсов - Qt Designer с использованием языка QML.

## Специальные требования
Программа должна обеспечивать взаимодействие с пользователем (оператором) посредством графического пользовательского интерфейса, разработанного согласно рекомендациям Заказчика. Программа должна обеспечивать удобный и быстрый просмотр необходимой информации.

# Стадии и этапы разработки
## Стадии разработки
Разработка должна быть проведена в три стадии:
1.	разработка технического задания;
2.	рабочее проектирование;
3.	внедрение.

## Этапы разработки
На стадии разработки технического задания должен быть выполнен этап разработки, согласования и утверждения настоящего технического задания.

На стадии рабочего проектирования должны быть выполнены перечисленные ниже этапы работ:
1. разработка программы;
2. разработка программной документации;
3. испытания программы.

На стадии внедрения осуществляется приемка програмного продукта Заказчиком и, при необходимости, пусконаладочные работы в целевой среде функционирования.
